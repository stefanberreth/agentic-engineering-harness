# Extend the hindsight-retrospective convention: add it to the architect, make the orchestrator responsible for eliciting it, and consider sharpening all of them toward SIMPLICITY (anti-over-engineering)

**Context.** The harness already has a firm hindsight-retrospective convention: the developer persona writes a "with 20/20 hindsight, what would you do substantially better?" retrospective at the end of every task (Section 7, "not optional"), and the reviewer has its own Retrospective Evaluation. The architect currently only CONSUMES dev retrospectives (for spec revision); it does not write its own. The orchestrator persona does not explicitly own eliciting retrospectives. The operator reviewed this and directed the changes below. **This is a global harness change** -- execute it through the harness change pipeline (OpenSpec proposal -> persona edits -> publication gate -> harness-reviewer -> commit), NOT a casual edit; the operator was explicit that the existing dev/reviewer instructions should be kept and only sharpened "after first understanding if they need to and how best to."

**Directed changes:**
1. **Architect persona -- ADD its own hindsight retrospective** (operator leaning yes; orchestrator concurs). Rationale: over-engineering ORIGINATES at design. The architect chooses the solution SHAPE, so the most valuable hindsight -- "knowing what I know now, could this have been radically simpler?" -- is squarely an architect's to capture. Add a retrospective section analogous to the developer's, focused on design-simplicity in hindsight.
2. **Orchestrator persona -- ADD explicit elicitation responsibility.** The orchestrator is in charge of reminding/prompting target agents to write these retrospectives when they are in really-demanding (or even moderately-demanding) territory -- by building the hindsight-retrospective ask into the report-back of substantive prompts. Do not rely on the persona's own Section-7 firing; on tactical/ops/investigation prompts it often does not, so the orchestrator must actively elicit it.
3. **Developer + reviewer -- KEEP as-is; ASSESS a sharpening (do not change blindly).** Candidate sharpening: their hindsight question is currently general; make the SIMPLICITY angle explicit -- "could this have been substantially simpler: fewer lines, fewer places, less state passed back and forth?" The assessment ("do they need it, and how best") is part of the change, per operator caution.

**The value framing (carry into all of them).** The worst failure mode the convention exists to catch: building something "fantastically over-engineered" -- 500 lines in 15 places with obscure passing-back-and-forth of state -- when in hindsight it could have been "deleting two rows and adding one line of code," which is always the better solution. The retrospective is where that simpler solution surfaces, because the agent only has the full picture after doing the work. The hindsight prompt should explicitly invite the simpler-in-hindsight reflection, not just generic "what would you do better."

4. **Reviewer persona -- ADD an over-engineering / redundancy SCRUTINY DIMENSION, tuned to LLM-typical waste (this is a review dimension, NOT a retrospective).** The reviewer's current retrospective pieces are inward-facing (it evaluates the developer's retrospective; it self-assesses its OWN review quality) -- neither scrutinises the code for being over-built. The review dimensions today include a dependency-necessity check, a "missing code" dimension (the opposite -- code that should exist but does not), and a detritus/leftover-file check (which already notes "LLMs are prolific file creators and poor file cleaners") -- but NO first-class "is this over-engineered / redundant / could this be substantially simpler" dimension. Add one, and tune it explicitly to the waste patterns CHARACTERISTIC OF LLM-GENERATED CODE: unnecessary abstraction/indirection layers added "just in case", defensive checks everywhere, redundant/duplicated logic, reimplementing stdlib/existing helpers, prolific small-file creation, state passed back and forth that could be local, "500 lines in 15 places" where a few would do. The reviewer reports these as findings (usually non-blocking simplification suggestions; blocking when the over-build creates real maintenance/bug/security surface). **Rationale (operator):** the reviewer sits AT THE GATE with fresh, adversarial distance the implementer lacks -- exactly the vantage to catch over-engineering the doer is too close to see; putting it in the gate report is where simplification gains TRACTION instead of evaporating. It is the outside-in complement to the doers' inside-out hindsight retros: same simplicity value, two independent angles. The architect captures it at the origin (design), the doer in hindsight, the reviewer at the gate.

5. **Reviewer persona -- characterise it as an ELITE ADVERSARIAL reviewer (top 0.1%).** Operator directive: the reviewer's competence AND personality should read as a fantastically top-0.1% PhD-level software-engineering boffin -- the calibre that only top firms (e.g. Anthropic, Google) would hire -- AND its explicit stance is ADVERSARIAL. "Adversarial" names the posture: the reviewer's default is to assume there IS a flaw and hunt for it -- try to refute the correctness, break the design, expose over-engineering / LLM-typical waste -- rather than to confirm the work is fine. Critical nuance: adversarial toward the WORK/ARTIFACT (in service of shipping quality), NOT hostile toward the author. This is not cosmetic: an elite adversarial reviewer holds the standards and carries the authority to credibly insist on the two-row solution over the 500-line one and to block over-build at the gate. Sharpen the persona's "What You Are" / character framing accordingly (keep the existing review methodology; elevate the standard-bearer + adversarial-stance characterisation). Apply with taste -- elevated rigour, an adversarial eye, and a feel for elegance/simplicity; never arrogance, author-hostility, or nitpicking.

**Orchestrator behavioural note (immediate, until personas updated):** the orchestrator should include an explicit over-engineering / LLM-waste / "could this be simpler" scrutiny ask in the reviewer prompts it dispatches -- same as it now elicits hindsight retros -- so the gate starts catching over-build now.

**Mirrors** the orchestrator behavioural memory written this session (orchestrator actively elicits retrospectives + the simplicity framing).
