# Container default git identity ("AEH Sandbox Agent") leaks into every commit; onboarding never overrides it

## Trigger

Operator opened GitLab statistics and saw "AEH Sandbox Agent <sandbox-agent@aeh.local>" credited as author on 667+ commits across the three repos this container has been touching (harness public, targets private, a target private). Expected to see their human identity. Caught the gap 2026-06-01 during a dep-bump push iteration on a target.

## The defect

The AEH harness container ships with a hardcoded git global identity:
- `user.name = AEH Sandbox Agent`
- `user.email = sandbox-agent@aeh.local`

This default is never overridden by any onboarding step. No per-repo override exists in the harness repo, the targets-private repo, or any target tree. Every commit -- whether produced by an orchestrator-dispatched agent or by the operator typing `git commit` directly inside the container -- is attributed to "AEH Sandbox Agent" until the operator manually runs `git config --global user.{name,email}` to override.

This directly violates `CLAUDE.md` § "Working Rules" rule about AI attribution:

> **No AI attribution in commits or files.** Never add `Co-Authored-By`, `Generated by`, or any other marker that identifies Claude, an LLM, an AI agent, or any automated tool as author or co-author. ... Commits are authored by the human; the tooling is invisible.

The current default is the exact opposite -- the AI-tooling identity IS the author field. The harness preaches the rule, then sets up an environment that violates it by default.

## Why this matters

- **Audit trail integrity.** Commits are not properly attributed to the human who authorised them. Future reviewers cannot run `git log --author=<operator>` to filter for human-authored commits because all commits read as the agent.
- **GitLab statistics distortion.** Contributors-graph, commit-count-per-author, project-stats panels all credit a placeholder identity rather than the actual contributor. Affects external perception of project contribution if any repo is public, and operator's own visibility into their work patterns.
- **Cross-environment portability assumption.** Operators using non-Docker setups, or multiple containers, or shared infrastructure, may legitimately want different identities per environment. The current default makes the wrong choice for everyone (no human's identity).
- **Discovery is reactive.** The defect was caught after 667+ commits accumulated -- only when the operator happened to check GitLab statistics. The onboarding sessions across multiple targets did not catch it.

## Proposed remedies (intake, not implementation)

1. **Onboarding gate.** `templates/playbooks/onboarding.md` Phase 0 (or earliest reasonable phase, before any commit happens) should detect the container default identity and require operator to set `user.name` + `user.email` via `git config --global` before any harness-side or target-side commit is made. Concrete check: `git config --global --get user.email` matches the sandbox-default pattern -> halt and prompt for real identity.

2. **Health-check.** `templates/playbooks/health-check.md` should verify the container identity is non-default. LOW finding if default identity still present (no actual commits affected if it's a fresh container); HIGH finding if `git log --author=...` shows recent commits attributed to the default identity (real attribution damage already done).

3. **Container provisioning.** If the harness container image is operator-managed, the image-build step that sets the global config should either (a) leave it unset (forcing first-use to set it) or (b) read identity from a passthrough environment variable (`AEH_OPERATOR_NAME` + `AEH_OPERATOR_EMAIL`) following the sandbox-env-provisioning pattern at `templates/tools/sandbox-env-provisioning.md`. The current "set a placeholder" choice is the wrong middle.

4. **Documentation cross-reference.** The CLAUDE.md "No AI attribution in commits or files" rule should include an explicit footnote: "this also forbids the author/email fields of commits identifying the tooling environment (e.g. 'AEH Sandbox Agent', 'Claude Code', etc.). Configure git identity to the operator's human identity at container first-use."

5. **Multi-container parity.** Each container (parallel orchestrator sessions per `cross-container-isolation`) needs its own identity setup -- the rule applies per container, not per repo, since git config global is container-private. Onboarding-gate should re-fire on every fresh container, not just first-target-onboarded.

## Out of scope (explicit)

- Historical rewrite. Operator's stated policy 2026-06-01: accept history as-is. Going-forward fix only.
- Per-commit AI-attribution detection (Co-Authored-By trailers etc.) is already covered by the existing CLAUDE.md rule + `feedback_no_ai_attribution`-equivalent memory. This intake is specifically about the author/email field defaults.
