## TL;DR

A practitioner's guide to using Claude Code as a managed team of AI personas (analyst, architect, developer, reviewer) rather than a "vibe coding" free-for-all. The core thesis: **you are no longer a programmer -- you are the micromanager of a team of brilliant but also very stupid experts.** The talk covers `CLAUDE.md` configuration, system prompts for distinct roles, context window management, OpenSpec for task/change tracking, and the minimal set of MCP servers worth keeping.

---

## 1. Why Claude Code

- Terminal-first workflow: `cd` into a directory, type `claude`, and go. No IDE sidebar -- the whole screen is the conversation.
- Low-level and configurable via markdown files; other agents (Gemini, Copilot) have heavier, more opinionated system prompts.
- Claude Code is Anthropic's primary monetisation path (Claude Max at $200/month), which drives ecosystem investment.
- Ecosystem: large community, "awesome-claude-code" repos, standards that other tools then imitate (commands, skills).

> "It's not just a website you go to like Google. It's a little spirit or ghost that lives on your computer." -- Andrej Karpathy (quoted in talk)

## 2. CLAUDE.md -- The Foundation

### What it is

A markdown instruction file that Claude Code reads on startup. Multiple levels:

| Location | Scope |
|---|---|
| `~/.claude/CLAUDE.md` | Global -- applies to all projects |
| `<project>/CLAUDE.md` | Project-specific instructions |
| `<project>/.claude/` directory | Team-shared config (commit to source control!) |

### What to put in it

- Build / dev / test commands
- Code style rules (or point at `.editorconfig`)
- Workflow rules ("always do TDD", "always plan before coding", "ask clarifying questions")
- Architecture decision records (ADRs)
- Notes that Claude maintains itself (tell it to update its own `CLAUDE.md`)

### Tips

- Use `@` prefix to link to other files from within markdown files -- acts like anchor links, tells Claude to read and consider the linked content.
- Despite starting with `.`, the `.claude/` directory **should be committed to source control**.
- Encode your team's onboarding guide into `CLAUDE.md` -- everything a new team member would need to know.
- `agents.md` is an emerging cross-tool standard (Claude, Gemini, etc. all look for it).

## 3. Essential Commands & Context Management

| Command | Purpose |
|---|---|
| `/init` | Create initial `CLAUDE.md` scaffold |
| `/context` | Show current token usage breakdown (system prompt, MCP tools, memory, conversation) |
| `/clear` | Clear conversation history to free context |
| `/compact` | Summarise and compress context (auto-triggers near limit) |
| `/skills` | List available skills |
| `/exit` or Ctrl+C | Exit session |

### Context window discipline

- **200k token window**, but treat **100k as a soft ceiling**.
- Run `/context` frequently to monitor usage.
- Past 100k tokens, Claude becomes less effective ("holding too much state").
- Auto-compact triggers at 80-95%, but **prefer exiting and restarting** over `/clear` or `/compact`.
- Before exiting, tell Claude to record its progress.
- MCP servers eat context tokens just by being loaded -- 50-60 servers can consume 20k+ tokens before you even start.

### Model choice

- Opus 4.5 for best quality (Claude Max gives ~8 hours/day).
- Sonnet 4.5 is very good and more accessible on the $20/month plan (~4-6 hours/day).
- You can run multiple Opus sessions simultaneously on Max.

## 4. The Four-Persona Workflow

The core methodology. Each persona is a **system prompt file** loaded via:

```bash
claude --system-prompt-file <persona>.md
```

### 4.1 Analyst (requirements gathering)

- **Input:** Your idea, a few lines of description
- **Output:** `requirements.md`
- **Process:** Claude interviews you -- asks about business context, capabilities, platform, constraints, success criteria
- The system prompt is itself generated by Claude (via Claude Desktop), tailored per project
- Spend 1-2 hours answering questions; aim to eliminate all ambiguity
- The `requirements.md` should give the architect everything needed

### 4.2 Architect (solution design)

- **Input:** `requirements.md`
- **Output:** `spec.md`
- **Process:** Designs the solution collaboratively -- platforms, languages, components, implementation phases
- System prompt is written *after* requirements are done (may be influenced by what came up)
- Can be specialised: "you are an AWS expert", "you are a Swift UI expert"
- `spec.md` includes phased breakdown: e.g. Phase 1: Python gRPC server, Phase 2: .NET client library, Phase 3: Unity app
- For cross-platform: fork the spec into platform-specific copies (one for iOS/Swift, one for Android/Kotlin)

### 4.3 Developer (implementation)

- **Input:** `spec.md`, tasks from OpenSpec
- **Output:** Working code on feature branches
- **Process:** Test-driven development, small incremental changes
- **Key rules in the developer system prompt:**
  - Write tests first, then make them pass
  - Comment code meaningfully (explain *what* and *why*)
  - Create a git branch for each task/component
  - Work in the **smallest increments possible**
  - Commit and push each branch
  - Coverage cannot go down
- If Claude makes a mess: `git reset --hard HEAD`, quit, restart, try again
- Think of it as "autocomplete" -- but at the level of a class, a controller, a set of views

### 4.4 Reviewer (code review)

- **Input:** Git branch/commit from the developer
- **Output:** `comments.md`
- **Process:** Uses `git log` and `git diff` to analyse changes; notes issues, edge cases, style violations
- Can be as picky as you want (spelling, brace style, etc.)
- **Workflow loop:**
  1. Developer commits on branch
  2. Exit Claude, restart with reviewer prompt
  3. Reviewer writes `comments.md`
  4. Exit Claude, restart with developer prompt
  5. Developer reads `comments.md`, addresses issues
  6. Repeat until satisfied
  7. Merge to main/dev

### Additional personas to consider

- **Designer** -- colour theory, UX patterns
- **Accessibility reviewer** -- WCAG compliance, screen reader testing

## 5. OpenSpec

**Website:** https://openspec.dev/
**Install:** `npm install -g @anthropic/openspec` *(name may have changed)*
**Init:** `openspec init` (select "Claude Code")

### What it does

- Replaces CC Sessions and several other MCP servers
- **Not an MCP server** (good -- doesn't eat context tokens)
- Integrates via commands and skills
- Enforces the discipline of breaking work into small, trackable changes

### Directory structure (post-v2, Feb 2026)

```
.claude/
  commands/
    opsx/
      apply.md
      ...
  skills/
    opsx/
      apply/
        skill.md
      ...
specs/
  changes/
    add-auth/
      ...
  change-archive/
  config.yaml
```

### Key concepts

| Concept | Description |
|---|---|
| **Proposal** | The requirements/idea (from analyst phase) |
| **Design** | The architectural spec (from architect phase) |
| **Tasks** | Checklist in `tasks.md` with checkboxes tracking completion |
| **Changes** | Individual features/modifications (e.g. "add-auth", "add-dark-theme") |
| **Commands** | Explicit, structured instructions (arguments, tags, steps) -- e.g. `/opsx apply add-auth` |
| **Skills** | Conceptual, abstract capabilities -- markdown files in `.claude/skills/` |

## 6. Extensions: MCP Servers, Commands & Skills

### MCP Servers (keep minimal)

Only two recommended:

| Server | Purpose |
|---|---|
| **Context7** (https://context7.com/) | Gateway to documentation for hundreds of frameworks, languages, and versions. Claude uses it to look up docs on demand. |
| **Serena** (https://oraios.github.io/serena) | LSP integration. Uses language servers (e.g. Roslyn for .NET) for symbol lookup, rename, etc. Much faster and lower-context than grep-based approaches. |

### Commands vs Skills

| | Commands | Skills |
|---|---|---|
| **Location** | `.claude/commands/` | `.claude/skills/` |
| **Nature** | Explicit, structured (arguments, tags, steps) | Conceptual, abstract (guidance, documentation) |
| **Invocation** | Slash commands (e.g. `/opsx apply add-auth`) | Referenced by Claude when relevant |
| **Use case** | Specific actions | Broader capabilities, replacing many MCP servers |

### Skills can include

- `skill.md` (core instructions)
- Documentation files, tutorials
- Schemas, scripts, reference material (e.g. PDF docs for Bluetooth spec)
- Skills don't fill up context with endpoint documentation the way MCP servers do

## 7. Key Principles

1. **You are not a programmer anymore** -- you are a micromanager of AI experts
2. **No vibe coding** -- always go through requirements, architecture, implementation, review
3. **Small increments** -- one feature, one branch, one reviewable PR
4. **Context discipline** -- stay under 100k tokens, exit and restart rather than compact
5. **System prompts per role** -- don't use one generic prompt; specialise per phase and per project
6. **Use Claude to write system prompts** -- LLMs are excellent at writing prompts for LLMs
7. **Cross-language fluency** -- if you know 2-3 languages, you can read all of them; let the architect pick the best tool for the job
8. **Git hygiene** -- branch per task, commit, push; `git reset --hard HEAD` if it makes a mess
9. **Review your own AI's code** -- treat it exactly like reviewing a junior developer's PR
10. **Minimal MCP servers** -- Context7 + Serena; replace the rest with skills/commands
